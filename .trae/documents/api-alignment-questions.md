**问题**: 后端当前使用的统一响应格式是什么？

**选项**:

- A. `{ code: 0, data: {}, message: "" }` - 标准包装格式

- C. 直接返回数据，无包装
- D. 其他（请说明）

**前端期望**:

```json
{
```

**后端结论**: ✅ 同意按前端期望的格式调整

- 所有接口将统一使用 `{ code, message, data }` 格式
- 错误码 0 表示成功，其他值表示错误

### 3. projectId 传递方式

**问题**: 工作台接口的 `projectId` 如何传递？

**选项**:

- A. 从 URL query 参数获取：`GET /workspace/stats?projectId=xxx`
- B. 从 HTTP Header 获取：`X-Project-Id: xxx`
- C. 从 JWT token 中解析当前项目
- D. 从 session 中获取

**前端期望**: **选项 A**（URL query 参数）

**理由**:

- 用户可能在多个项目间切换
- URL 参数便于调试和分享
- 符合 RESTful 设计

**后端结论**: ✅ 同意选项 A

- 后端将使用 URL query 参数传递 projectId
- 例如：`GET /api/v1/workspace/stats?projectId=xxx`

**待确认**: 无

## 三、待前端确认问题汇总

### 问题 5：任务的优先级如何确定？

**前端回答**：

**MVP 阶段**：优先级可以**后端默认设置**，不需要前端设置界面。

**优先级规则建议**：

```
HIGH（高优先级）:
- 距离截止日期 < 3天
- 或 Key 被标记为 critical

MEDIUM（中优先级）:
- 距离截止日期 3-7天
- 默认优先级

LOW（低优先级）:
- 距离截止日期 > 7天
- 或没有截止日期
```

**后续迭代**：可以增加优先级设置功能，让项目管理员调整规则。

### 问题 11：`pending` 和 `completed` 对应哪些 Translation 状态？

**前端回答**：

| 审核筛选状态 | 对应的 Translation 状态           |
| ------------ | --------------------------------- |
| `pending`    | `REVIEWING`（待审核）             |
| `completed`  | `APPROVED` + `REJECTED`（已审核） |

**说明**：

- 审核员视角的 "pending" = 等待我审核的翻译 = `REVIEWING` 状态
- 审核员视角的 "completed" = 我已经处理完的翻译 = `APPROVED` 或 `REJECTED`
