# 用户流程与待办（MVP → 可用闭环）

本文件用于把「用户从 0 到 1 使用 Lingux 的关键路径」与「按优先级拆分的待办」沉淀到仓库中，作为后续迭代的对齐基线。

## 角色与职责

- 管理员（Admin）：创建项目、配置语言、发布/回滚、权限与规则配置
- 内容/译员（Editor/Translator）：录入 Key、编辑翻译内容、处理退回意见
- 审核员（Reviewer）：审核翻译（通过/退回），把控质量
- 开发者（Developer）：接入消费端 SDK/拉取静态 JSON，协助导入/导出与自动化

## 核心对象（概念模型）

- Project：项目空间，定义 baseLocale（源语言）与启用的目标语言集合
- Locale：系统语言库（如 zh-CN / en-US），Project 通过关联启用
- Namespace：项目内按模块分组（common/auth/home）
- Key：文案标识（例如 common.button.submit），归属 Namespace，类型可为 TEXT/RICH_TEXT/ASSET
- Translation：某个 Key 在某个 Locale 下的内容与状态
- Release（待实现）：一次发布产物的版本号、范围（页面/命名空间/全量）、产物地址与回滚点

## 状态流转（翻译工作流）

当前建议的翻译状态机（与后端枚举保持一致）：

- PENDING：待翻译/待补全
- TRANSLATING：自动翻译中（LLM/供应商任务执行期）
- REVIEWING：待审核
- APPROVED：已审核通过（可进入发布）
- PUBLISHED：已发布（消费端可见）

典型规则：

- 编辑翻译内容后：状态回到 PENDING 或 REVIEWING（取决于是否需要再次审核）
- 只有 APPROVED 才允许进入 PUBLISHED（发布动作强约束）

## 用户流程（User Journey）

### 1) 项目初始化（Admin/Developer）

1. 创建 Project（填写名称、描述、baseLocale）
2. 从系统 Locale 库中选择并启用目标语言（Project Locales）
3. 建立命名空间（Namespaces）
4.（可选）导入现有 Key（JSON/CSV/Excel/CLI）

验收标准：

- 新项目可见、可进入项目工作区
- 项目已启用语言在 “Locales” 中可维护（增/删）
- 至少一个命名空间可用于录入 Key

### 2) 录入词条（Editor/Developer）

1. 进入项目工作区 → Keys
2. 选择命名空间
3. 新建 Key（name/description/type）
4.（可选）录入源语言内容（baseLocale 的 Translation）

验收标准：

- Key 在列表可见，且能看到项目启用语言的翻译占位状态
- Key 的重复约束（同 namespace 内 name 唯一）有明确提示

### 3) 翻译（Translator/AI）

路径 A（人工）：

1. 在 Keys 列表筛选 PENDING
2. 打开翻译编辑面板（Drawer/Modal）
3. 为目标语言补全翻译
4. 提交进入 REVIEWING（或保持 PENDING 等待二次处理）

路径 B（LLM 批量）：

1. 选择命名空间/Key 范围
2. 触发 LLM 翻译任务（异步）
3. 状态从 PENDING → TRANSLATING → REVIEWING（或直接 PENDING 视策略）

验收标准：

- 单条翻译可编辑并保存
- 批量翻译可观察进度与结果落库（需要任务与队列能力）

### 4) 审核（Reviewer）

1. 筛选 REVIEWING 的 Key/语言
2. 查看差异与上下文（截图/页面路径/变量示例/TM 建议）
3. 通过 → APPROVED，或退回 → PENDING（附带原因）

验收标准：

- 审核动作可追溯（审计日志/操作人/时间）
- 退回必须包含原因（用于闭环）

### 5) 发布与回滚（Admin）

1. 选择发布范围（全量 / 按命名空间 / 按页面路径）
2. 执行质量门禁（占位符/ICU/术语/敏感词/长度校验）
3. 生成静态 JSON 产物并发布（CDN/对象存储）
4. 出现问题 → 选择上一次 Release 一键回滚

验收标准：

- 发布动作可复现、可回滚
- 发布报告包含版本号、范围、语言、变更 Key 列表

## 待办清单（按优先级）

### P0：跑通“录-译-审-发-回滚”最小闭环

- 翻译编辑：支持按语言编辑并保存（单条/多条）
- 翻译状态流转：REVIEWING/APPROVED/PUBLISHED 的 UI 与权限约束
- 审核台：待审列表、批量通过/退回、退回原因
- 发布：生成静态 JSON（按项目/命名空间/页面路径）
- 回滚：基于 Release 版本切换回退
- 质量门禁：占位符/ICU 基础校验、敏感词拦截（阻断发布）

### P1：效率与规模化（让日常“更快”）

- 批量导入/导出：i18next JSON、CSV/Excel（含页面路径/资源路径）
- LLM 自动翻译：异步任务、重试、供应商切换、用量统计
- 翻译记忆（TM）与术语库：相似匹配、一键采纳、术语审批
- 截图上下文：Key 关联截图与页面路径，编辑器侧边联动展示

### P2：开发者体验（让接入“更稳”）

- API Key 管理：生成/吊销/权限范围
- 消费端拉取协议：版本、缓存策略、增量拉取
- SDK 示例与文档：React/Vue/Next.js，SSR/CSR 的用法差异
- Webhook/通知：发布完成、待审提醒、逾期提醒

### P3：企业级增强（可并行，不阻塞主线）

- 多租户隔离、组织与成员管理（SCIM）
- 审计与合规：操作留痕、导出、告警
- 移动端资源包：iOS .strings / Android .xml，OTA 差分发布

## 里程碑验收建议

- 里程碑 A（项目可用）：创建项目 → 启用语言 → 创建命名空间 → 新建 Key → 编辑翻译
- 里程碑 B（流程闭环）：翻译提交 → 审核通过/退回 → 发布 → 消费端拉取 → 回滚
