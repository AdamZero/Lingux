# 多语言管理平台（生产端） · 架构设计

## 1. 定义与边界

本文档聚焦于**数据生产端**，即**多语言管理平台**本身。它的核心职责是**生产、管理和发布**合格的多语言文案。

- **包含**：供内部团队（PM、开发、译员）使用的**管理后台前端**，以及为其提供服务的**核心后端**。
- **不包含**：最终消费多语言文案的业务网站（数据消费端）。
- **核心产出**：当发布操作被触发时，系统生成静态的多语言 JSON 文件，并将其推送到 CDN，供消费端使用。

## 2. 核心设计原则

- **前后端分离**：前端负责视图与交互，后端负责业务逻辑与数据，通过 API 通信，便于独立开发、部署与扩展。
- **模块化**：后端按业务领域（如项目、翻译、发布）划分模块，前端同步按功能组织页面与组件，降低耦合。
- **无状态服务**：后端 API 服务尽可能无状态，便于水平扩展与高可用部署。用户状态通过 Token（JWT）管理。
- **异步处理**：对于耗时操作（如批量机翻、发布打包），采用消息队列异步处理，避免阻塞 API，提升用户体验。

## 4. 前端架构设计 (管理平台前端)

- **技术栈**
  - **框架**: **React**。生态成熟，适合构建复杂的企业级应用。
  - **UI 组件库**: **Ant Design**。提供高质量、开箱即用的企业级组件，极大加速开发。
  - **状态管理**: **Zustand**。API 简洁，上手快，能满足绝大部分场景。
  - **数据请求**: **TanStack Query (React Query)**。优雅地管理服务端数据，自动处理缓存、重试、加载状态。
  - **路由**: **React Router**。React 官方推荐的路由库，稳定可靠。
  - **构建工具**: **Vite**。提供极速的开发体验。

- **模块划分 (按页面/功能)**
  - **`Dashboard`**: 概览页，展示翻译覆盖率、待办任务、成本统计等。
  - **`Project`**: 项目管理模块，包含项目列表、详情、命名空间、环境配置。
  - **`Workspace`**: 核心工作区，包含文案列表（支持按页面路径/命名空间/状态筛选）、翻译编辑器、上下文侧边栏。
  - **`Terminology`**: 术语库管理模块。
  - **`Publishing`**: 发布历史、版本对比、回滚操作。
  - **`Settings`**: 个人设置、团队成员管理、LLM 服务商配置。
  - **`Auth`**: 登录页，处理企业协作平台 OAuth 回调。

## 5. 后端架构设计 (平台核心后端)

- **技术栈**
  - **框架**: **NestJS (Node.js / TypeScript)**。提供高度结构化、模块化的架构，专为构建复杂、可维护的后端服务而设计。
  - **数据库**: **PostgreSQL**。功能强大的开源关系型数据库，支持 JSONB 字段。
  - **ORM**: **Prisma**。提供极致的类型安全，能根据数据库 schema 自动生成类型，提升开发效率。
  - **API 风格**: **RESTful API**。
  - **消息队列**: **BullMQ (基于 Redis)**。用于处理所有异步任务（批量机翻、发布打包、发送通知）。
  - **缓存**: **Redis**。用于缓存高频读取的数据（如 TM 匹配结果、用户权限）。

- **模块划分 (按领域)**
  - **`AuthModule`**: 处理用户认证（JWT）、OAuth 登录、角色权限（RBAC）。
  - **`ProjectModule`**: 管理项目、命名空间、环境、成员等核心实体。
  - **`KeyModule`**: 管理文案 Key、条目 Entry、上下文（页面路径、截图等）。
  - **`TranslationModule`**: 核心翻译服务，包含调用 LLM 的 Provider 子模块和处理异步任务的 Job 子模块。
  - **`QualityModule`**: 负责质量门禁校验逻辑（ICU、占位符、敏感词等）。
  - **`PublishingModule`**: 负责生成发布版本、打包资源、推送到 CDN、记录历史与回滚。
  - **`NotificationModule`**: 负责向企业协作平台异步发送通知。

## 6. DevOps 与基础设施

- **容器化**: 所有服务使用 **Docker** 打包。
- **编排**: 开发环境使用 **Docker Compose** 一键启动；生产环境推荐使用 **Kubernetes (K8s)**。
- **CI/CD**: 使用 **GitHub Actions** 或 **GitLab CI**，在 `main` 分支合并时自动测试、构建、部署。
- **托管**:
  - **前端静态资源**: 部署到 **CDN**（如 AWS S3 + CloudFront）。
  - **后端服务**: 部署在云服务器或容器平台。
  - **发布产物 (JSON 文件)**: 推送到 **CDN**。

## 7. 架构选型理由 (分离架构 vs. Next.js)

选择 `React (Vite) + NestJS` 的前后端分离架构，而非统一的 `Next.js` 应用，主要基于三点：

1.  **业务重心匹配**：本平台是**重逻辑、重数据**的内部管理后台 (SaaS)，而非内容展示型网站。分离架构更能发挥各自优势，Next.js 的 SSR/SSG 优势在此场景下无法体现。
2.  **后端结构化需求**：NestJS 提供的**依赖注入、模块化、分层架构**等特性，专为构建复杂、可维护的后端服务而设计，比 Next.js 的 API Routes 更适合承载本平台复杂的业务逻辑。
3.  **大量异步任务处理**：本平台核心功能涉及大量长耗时任务（如批量翻译）。NestJS 作为持久化运行的服务器进程，与**消息队列**是天作之合，能直接、高效地处理后台任务，而 Next.js 的 Serverless 特性不擅长此场景。
